§1 — Five subsystems → five regions

Each harness subsystem maps to a fixed screen region. Users build muscle memory: Instructions → left gutter, State → scroll area, Verification → bottom dock, Scope → composer, Lifecycle → overlays.

fixed regions
§2 rule 1 — Bounded re-layout

Only the middle ChatLog region scrolls. Header, footer, composer, and status bar are fixed-height across the entire session. Nothing else may resize mid-loop.

1fr scroll · fixed dock
§3 — Ergonomic layout grid

Six-region vertical stack: Header (1 line, dock top), ChatLog (flex 1fr), StatusBar (1 line, fixed), Composer (3–8 lines, soft-wrap). Two stable eye anchors at bottom.

6-region stack
§3 — Symmetric horizontal margins

2 columns each side. Symmetry reads as calm; asymmetric margins read as active. The 2-col value is the "book width" — the eye-rest zone.

2-col margin
§4 — #chrome is a single docked container

Status bar and composer never separate. They are one logical "bottom of the screen" zone — the bottom-anchor muscle memory zone.

#chrome
§4.3 — Header: summary rail

Header aggregates Scope + State + Lifecycle in one glanceable line. 1 line collapsed, docked top. Click expands to overlay panel without resizing ChatLog. Default-on.

§4.3 Header

§2 rule 2 — Quiet by default

Idle states must not blink, pulse, or redraw. Motion is reserved for live work. When the loop is between iterations, the screen is a held still image with only the cursor in the composer.

no animation
§2 rule 3 — One anchor per iteration

Each loop iteration produces one spatial anchor: a ▎ label, a tool marker, or a user message. Multiple anchors per iteration = layout fights itself.

one label per turn
§2 rule 5 — Indentation encodes nesting

Tool output is indented 2 cols right of its marker. Thinking body is indented 2 cols right of the assistant label. User/assistant message bodies share the same left edge as their label.

2-level indent
§4 — ThinkingDisplay · CollapsibleToolOutput

Thinking body: indented under label, collapsible via marker click. Tool output: indented under marker, max 20 lines with scroll. Both use the second-tier indentation.

collapsible bodies
§4 — StatusBar is glanceable, 1 line

The always-on index: model, turns, tools, context ratio. Hard-capped at 1 line. This is Tier 1 of the progressive disclosure model — always visible.

Tier 1
§5 — Progressive disclosure: 3 tiers

Tier 1: StatusBar (always). Tier 2: marker line + ▎ label (visible when scrolled to). Tier 3: collapsed body / modal (click to load). Never flatten all three.

3-tier model

§6 — ThinkingMarker motion

Spinner ticks during thinking; flips to "thought · Ns" once complete. The marker is the only animated element in the idle-to-active transition.

spinner tick
§4 — StreamingOverlay

Inline, replaces AssistantMessage until final. Content updates in place — no slide, no fade. Token delta triggers a live update of the same DOM element.

in-place update
§6 — ToolCallMarker glyph swap

Marker swaps from ⊙ running → ⊙ done (or ⊗ error) on tool_use complete. Instant — no easing, no transition.

instant swap
§2 rule 4 — Predictable monotonic scroll

New content always appears at the bottom of the scroll region and pushes older content up. Never insert in the middle. Never re-order past content.

bottom-up scroll
§5 — No "pending" state indicator

The TUI correctly jumps from "user sent" → "thinking spinner" → "streaming". A pending placeholder row would be dead pixels most of the time.

skip pending
§6 — Instant transitions

Easing accumulates into perceptible lag in long sessions. Instant transitions read as "the UI is recording what I did" — the long-loop aesthetic in motion form.

no easing

PermissionScreen Overlay

A hard gate: the loop pauses, the screen is replaced, and the user must consent. The thick red border is not aesthetics — it is a safety signal. The rest of the layout yields entirely.

§1 — Lifecycle: full-screen overlays

PermissionScreen and ToolCallModal are the Lifecycle subsystem's screen region. Hard-interrupt moments: consent and deep-dive. The rest of the layout yields.

Lifecycle region
§2 rule 6 — Hard interrupts fill the screen

Lifecycle overlays take the full viewport with a thick border and centered alignment. They never slide in from an edge — they replace the layout entirely.

full-screen replace
§4 — PermissionScreen is a hard gate

70% width, thick red border. The red border is not aesthetics — it is a signal: "the loop is paused." If permission prompts look like transient toasts, users dismiss them reflexively and the safety guarantee weakens.

safety signal
§6 — Appears instantly

PermissionScreen appears instantly, replaces layout. Triggered by hook pre-tool. No animation, no slide-in — instant replacement is the long-loop aesthetic applied to consent.

instant replace
§5 — Hard interrupts for consent only

Tool call errors (soft interrupts) should NOT go full-screen — they become a SystemNote inline. Reserve full-screen for consent only. This distinction preserves the severity hierarchy.

consent ≠ errors

§4 — ToolCallModal: 80% × 80%

Center, full-screen overlay. 80% width, 80% height. Modal — view-only deep dive. Double-click a ToolCallMarker to open; Escape or Close button to dismiss.

80% × 80%
§2 rule 6 — Hard interrupts fill the screen

Like PermissionScreen, ToolCallModal replaces the layout entirely. The loop is paused during viewing. This is the "deep-dive" lifecycle interrupt.

full-screen replace
§6 — Appears instantly

Triggered by double-click on marker. No slide, no fade, no easing. Appears instantly — the same instant-replace pattern as PermissionScreen.

instant replace
§5 — Progressive disclosure: Tier 3

Tier 1: StatusBar (always). Tier 2: marker line (visible). Tier 3: modal body (click to load). This modal is the deepest tier — full args + result markdown.

Tier 3 deep-dive
§5 — On-demand detail, not always-on

Full detail (per-tool call, full message history) is on-demand behind click-to-expand. The 1-line StatusBar cap is not negotiable — it keeps Tier 1 glanceable.

on-demand

§1 — Header (summary rail)

Header aggregates Scope + State + Lifecycle in one glanceable line. It is the sixth fixed region, docked at the top of the viewport.

§1 summary rail
§2 rule 1 — Bounded re-layout

Header is exactly 1 line in collapsed state. It never grows, regardless of how many MCPs or todos exist. The ChatLog remains flex 1fr.

1 line · never grows
§2 rule 2 — Quiet by default

Collapsed Header is the glance density ceiling — even idle, it shows the session's load shape in one line, but it must never grow taller when collapsed.

glance density ceiling
§4.3 — Aggregate indicators reflect worst state

MCP: ● all healthy / ◌ any error / ○ all disabled. Todo: ◐ has active / ✓ all done / ○ empty. Subagent: ◐ has running; hidden when count = 0.

worst-state aggregates
§4.3 — Hide rules: zero state

If any section's count is zero (MCP = 0, todo = 0, subagent = 0), that section disappears from the collapsed line. Empty sections do not occupy visual space.

zero → hide
§5 — Memory pattern applied to layout

Collapsed Header is the bounded always-on index (3 sub-buttons sharing 1 line). Each per-section overlay is its own bounded on-demand topic. The index never grows; the topic loads only when the user asks.

index + per-section topic
§4.3 — Per-section buttons (3 click targets)

Each of the 3 buttons is an independent click target. The collapsed line is NOT a single click target — clicking the MCP button only expands MCP (see State 7). Buttons have hover styling (text accent + font-weight: 500) to signal clickability.

3 buttons · independent
§4.3 — No ▼ prefix glyph

The original 2026-06-19 spec used a ▼ glyph as a "click to expand" affordance. In the 2026-06-20 redesign, each button IS its own affordance — the glyph + label + count is the click target itself, no extra glyph needed.

no extra glyph

§4.3 — Per-section overlay (MCP only)

Clicking the MCP button expands ONLY the MCP overlay. Todo and Subagent buttons stay in collapsed state — mutual exclusion invariant. Each overlay shows exactly one section's detail rows.

mutual exclusion
§4.3 — Active button indicator

The MCP button shows an "active" state: bg highlight + 2px accent border-bottom + bold text. This signals which section's overlay is currently open. Clicking the same button again collapses (see State 6 for the collapse-to-collapsed behavior).

2px accent border
§4.3 — Expanded state: overlay, not resize

ChatLog geometry is unchanged. The expanded panel overlays the top of the ChatLog — visually obscured only, not reflowed. Scroll position is preserved when collapsed/switched.

overlay · not resize
§4.3 — Per-section overlay IDs

The overlay is mounted with id header-overlay-mcp (and similarly header-overlay-todo, header-overlay-subagent) to avoid DuplicateIds when switching (the old overlay may still be in the DOM pending async removal).

per-section ID
§4.3 — Three indent levels maximum

Outer column: section header (glyph + label:N/M). 2-col right: detail rows (`<glyph> <name>`). No fourth tier. This matches the ChatLog's two-level indentation convention.

3 levels max
§6 — Instant replace

No animation, no slide. Click = instant toggle (mount/remove). ESC collapses. The overlay never appears without explicit user action — no auto-expand on errors.

instant toggle

§4.3 — Switch case (3-way toggle)

Clicking the Todo button while MCP overlay was open triggers the "switch" case: existing MCP overlay removed, Todo overlay mounted. The Todo button is now active; MCP returned to collapsed state.

switch (no overlap)
§4.3 — Mutual exclusion invariant

At most ONE HeaderOverlay is mounted at any time. This prevents UI crowding in long sessions and matches §2 rule 2 (quiet by default — only one region of motion at a time).

at most 1 overlay
§4.3 — ChatLog scroll position preserved

When switching sections, the ChatLog geometry and scroll position are preserved. The overlay swaps content but doesn't reflow the chat behind.

scroll preserved
§6 — Instant replace

No animation, no slide, no crossfade. Switching from MCP → Todo overlay is one frame: old removed, new mounted. Per spec §6, all transitions are instant.

instant swap
§5 — No auto-load

The switch is only triggered by an explicit user click. No auto-expand on errors. No auto-switch when state changes (e.g., a todo being marked active does NOT auto-open the Todo overlay).

no auto-load
§4.3 — 3 collapse mechanisms

Beyond clicking the same button (toggle off), the overlay also collapses on: (a) ESC key — ("escape", "collapse_header") BINDING; (b) click outside the overlay — chat log / status bar / composer (App.on_click handler).

ESC + click-outside

Layout Grid Reference

The canonical grid from §3, reproduced as an annotated diagram. The 6-region stack with region boundaries.

┌──────────────────────────────────────────────────────────────────────┐ │ ▼ ● MCP:3/3 ◐ 2/5 todos ◐ 1 subagent │ ← Header (1 line, dock top, collapsible) ├──────────────────────────────────────────────────────────────────────┤ │ │ │ ChatLog (flex, 1fr, scrolls) │ ← State subsystem │ ← left & right margins 2 cols → │ │ ← vertical padding 1 row top → │ │ │ │ ▎ you ····· user message body │ ← Instructions (gutter) │ ▎ assistant ··· thinking / response / tools │ │ │ ├──────────────────────────────────────────────────────────────────────┤ ← margin: 0 2 1 2 │ StatusBar (1 line, fixed, inside #chrome) │ ← Verification ├──────────────────────────────────────────────────────────────────────┤ │ Composer (3..8 lines, soft-wrap, grows, focused input) │ ← Scope └──────────────────────────────────────────────────────────────────────┘ ↑ top anchor — click expands Header overlay ↑ bottom anchor — caret never moves between sessions
§3 — Two stable eye anchors

Status bar (bottom edge) and composer caret (just above it). The eye can return to either without search. The chat log is read between these two anchors, top-down.

§3 — Soft-wrap composer, never horizontal scroll

Multi-line prompts are the user's thinking space; horizontal motion breaks the writing flow. Composer caps height at 8 lines — long prompts scroll inside, the rest of the layout never moves.

§4 — Inline children share left edge

All turn labels, messages, markers share the same left edge. Only second-tier elements (thinking, tool output) indent 2 cols right. This two-level indentation is the entire visual hierarchy — no third tier.
